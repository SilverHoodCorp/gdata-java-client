Google's additions to OAuth 1.0a authorization as specified in Google's OAuth API Reference (see package description for an overview).

Before using this library, you need to set up your application as follows:

  1. For web applications, you should first register your application at the registration page. You will be provided with two pieces of information you will need:
    • OAuth Consumer Key: use this for the this will be used as the {@code consumerKey} on every OAuth request, for example in {@link com.google.api.client.auth.oauth.AbstractOAuthRequest#consumerKey}.
    • OAuth Consumer Secret: use this for the {@link com.google.api.client.auth.oauth.OAuthHmacSigner#clientSharedSecret} when using the {@code "HMAC-SHA1"} signature method.
  2. For installed applications, unregistered web applications, or web application running on localhost, you must use the {@code "HMAC-SHA1"} signature method. Use {@code "anonymous"} for the {@code consumerKey} and {@code clientSharedSecret}.
  3. For the {@code "HMAC-SHA1"} signature method, use {@link com.google.api.client.auth.oauth.OAuthHmacSigner}.
  4. For the {@code "RSA-SHA1"} signature method, use {@link com.google.api.client.auth.oauth.OAuthRsaSigner}. See the instructions for generating a self-signing private key and public certificate .

After the set up has been completed, the typical application flow is:

  1. Request a temporary credentials token ("request token") that has not yet been authorized from the Google Authorization server using {@link com.google.api.client.googleapis.auth.oauth.GoogleOAuthGetRequestToken}. A callback URL must be specified for web applications, but does not need to be specified for installed applications.
  2. Direct the user to a Google Accounts web page to allow the end user to authorize the request token using using {@link com.google.api.client.googleapis.auth.oauth.GoogleOAuthAuthorizeTokenRequestUri}.
  3. After the user has granted the authorization:
    • For web applications, the user's browser will be redirected to the callback URL which may be parsed using {@link com.google.api.client.auth.oauth.OAuthAuthorizeTokenResponseUri}.
    • For installed applications, Google gives the user a verification code which the user must then type into the application.
  4. Request to exchange the request token for a long-lived access token from the Google Authorization server using {@link com.google.api.client.googleapis.auth.oauth.GoogleOAuthGetAccessToken}. This access token must be stored.
  5. Use the stored access token to authorize HTTP requests to protected resources in Google services by setting the {@link com.google.api.client.auth.oauth.OAuthAuthorizer} for {@link com.google.api.client.http.HttpHeaders#authorizer}.
  6. For 2-legged OAuth, use {@link com.google.api.client.googleapis.auth.oauth.Google2LeggedOAuthUriEntity} as the URI entity builder to set the requestor ID.
  7. To revoke an access token, use {@link com.google.api.client.googleapis.auth.oauth.GoogleOAuthRevokeToken}. Users can also manually revoke tokens from Google's change authorized websites page.
